This is downtime and we want to avoid that.重要的是要注意，当 Docker 容器更新时，它必须重建，并且将旧容器换成新容器的过程会产生时间间隔，在此期间没有容器运行。这是停机时间，我们希望避免这种情况。 A common way around this is to route traffic through some sort of reverse proxy network appliance. This could be NGINX or HAProxy, to name two popular options. Both are great and allow us to shape traffic independently of our container backend. 解决此问题的一种常见方法是通过某种反向代理网络设备路由流量。这可能是 NGINX 或 HAProxy ，仅举两个流行的选项。两者都很棒，让我们能够独立于容器后端来塑造流量。In an architecture with a proxy, we can build a new container, stand it up in parallel with the old, and then start sending traffic to this new container before tearing down the old. Once this is successfully completed, the inactive environment becomes the active environment and the process can begin anew.如果出现问题，回滚就像将流量指向原始环境一样简单。成功完成后，非活动环境将变为活动环境，并且该过程可以重新开始。 We can use a proxy to direct traffic to either a blue or green environment.
